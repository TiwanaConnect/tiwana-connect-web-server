# API Contract

The mobile app will consume routes under `/api/mobile/*`. Phase 0 only creates health endpoints and shared response conventions.

## Response Shape

Success:

```json
{
  "data": {},
  "error": null
}
```

Error:

```json
{
  "data": null,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Readable error message"
  }
}
```

## Phase 2 Family Tree APIs

Implemented mobile endpoints:

- `GET /api/mobile/members/tree`
- `GET /api/mobile/members/search?q=&limit=&cursor=`
- `GET /api/mobile/members/relationship?targetMemberId=`

Implemented admin endpoints:

- `GET /api/admin/family-tree`
- `GET /api/admin/family-tree/relationship`
- `GET /api/admin/family-tree/standalone`
- `GET /api/admin/family-tree/heads`

Privacy rule: the backend uses the full family graph internally, including hidden female members, for traversal and relationship finding. Mobile responses mask hidden female real names by setting `name` to `null` and using alias or `Private family member` as `displayName`. Admin responses can include real names.

## Bulk Import Notes

Member bulk import supports `isFamilyHead` as an optional boolean column.
Future work: add relationship mapping columns such as `fatherIdentifier`, `motherIdentifier`, and `spouseIdentifier` to the bulk member import flow after the single-create relationship UX is proven.

Member IDs are generated by Prisma with `cuid()` and are never entered manually in admin forms. Use IDs only as internal API references.

Admin screens show `fullName || alias || "Unnamed Member"` for all members, including hidden female members. Mobile APIs continue to mask hidden female names with alias or `Private family member`.

## Phase 3 Event APIs

Mobile:

- `GET /api/mobile/events`
- `GET /api/mobile/events/[id]`
- `POST /api/mobile/events`
- `POST /api/mobile/events/[id]/rsvp`
- `GET /api/mobile/events/mine`

Admin:

- `GET /api/admin/events`
- `POST /api/admin/events`
- `GET /api/admin/events/[id]`
- `PATCH /api/admin/events/[id]`
- `POST /api/admin/events/[id]/cancel`
- `POST /api/admin/events/[id]/pin`
- `POST /api/admin/events/[id]/unpin`
- `GET/POST /api/admin/events/[id]/invites`
- `DELETE /api/admin/events/[id]/invites/[inviteId]`

All-family events create normalized `EventInvite` records for active members. RSVP state lives on invite records. Mobile event creator/invitee display names must use privacy-safe member display rules.

Event invite audiences support all family, all males, all females, manual member selection, and branch placeholder selection. Invite records remain normalized.

Mobile:

- `GET /api/mobile/funds`
- `GET /api/mobile/funds/[id]`
- `POST /api/mobile/funds`
- `POST /api/mobile/funds/[id]/requests`
- `POST /api/mobile/funds/[id]/contributions`
- `GET /api/mobile/funds/my-requests`
- `GET /api/mobile/funds/my-contributions`

Admin:

- `GET /api/admin/funds`
- `POST /api/admin/funds`
- `GET /api/admin/funds/[id]`
- `PATCH /api/admin/funds/[id]`
- `POST /api/admin/funds/[id]/close`
- `POST /api/admin/funds/[id]/cancel`
- `POST /api/admin/funds/[id]/archive`
- `POST /api/admin/funds/[id]/transactions`
- `GET /api/admin/funds/transactions`
- `POST /api/admin/funds/transactions/[transactionId]/confirm`
- `POST /api/admin/funds/transactions/[transactionId]/reject`
- `POST /api/admin/funds/[id]/requests`
- `GET /api/admin/funds/requests`
- `PATCH /api/admin/funds/requests/[requestId]`
- `POST /api/admin/funds/requests/[requestId]/waive`
- `POST /api/admin/funds/requests/[requestId]/cancel`

Fund balances are derived from confirmed ledger rows only. Contribution, zakat income, sadaqah income, refund, and adjustment rows are credits. Expense and disbursement rows are debits. Amounts are stored as positive decimal values; transaction type determines direction. Mobile contribution submissions are recorded as pending until an admin confirms them.

Admin:

- `GET /api/admin/directory`
- `GET /api/admin/directory/[memberId]`
- `PATCH /api/admin/directory/[memberId]/settings`
- `GET/POST /api/admin/tags`
- `PATCH/DELETE /api/admin/tags/[id]`
- `GET/POST /api/admin/members/[id]/tags`
- `DELETE /api/admin/members/[id]/tags/[tagId]`
- `GET /api/admin/help-requests`
- `GET/PATCH /api/admin/help-requests/[id]`

Directory entries are derived from `Member` records. Mobile directory responses mask hidden female full names, exclude blocked/deleted members, respect `MemberDirectorySetting.visibility`, and show phone only when `showPhone=true`. Help requests are member-to-member workflow records only; no chat, media upload, or real notification provider is included.

## Push Notification APIs

Mobile:

- `POST /api/mobile/devices/register-push-token`
- `POST /api/mobile/devices/unregister-push-token`
- `GET /api/mobile/devices/push-tokens`
- `GET /api/mobile/notifications`
- `GET /api/mobile/notifications/unread-count`
- `POST /api/mobile/notifications/[id]/read`
- `POST /api/mobile/notifications/read-all`
- `POST /api/mobile/notifications/[id]/archive`
- `GET/PATCH /api/mobile/notification-preferences`

Admin:

- `GET /api/admin/notifications`
- `POST /api/admin/notifications/send-test-push`
- `GET /api/admin/notifications/stats`
- `GET /api/admin/push-deliveries`

Push delivery uses Expo Push Notifications only. Notification payloads include `type`, `entityType`, `entityId`, and `actionUrl` for mobile routing. In-app notifications are created even when `PUSH_ENABLED=false`; provider delivery is skipped in that mode.
